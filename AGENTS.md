## セッション開始

1. 現在の working directory がこの project root またはその配下であることを確認する。
2. mozyo-bridge の central preset rules を読む:
   - committed docs では portable 表記 `.mozyo-bridge/rules/presets/redmine-governed/agent-workflow.md` を使う。
   - runtime で実ファイルを読む際も `.mozyo-bridge/rules/presets/redmine-governed/agent-workflow.md` を読む。repo-local store (`.mozyo-bridge/rules/...`) の path は repo root からの相対でそのまま読める。central store の home prefix は `mozyo-bridge rules home --resolved` の出力で解決する (`--resolved` 出力は debug / runtime 用で、committed docs に貼らない)。
   - resolved path や central preset を読めない場合は、読んだふりをせず停止し、`mozyo-bridge rules install` 等の復旧を operator に求める。
3. project-local docs (例: target project の README、codebase 規約 docs、運用 docs) が存在する場合は読む。これらの path は target project 側の任意の convention であり、mozyo-bridge の runtime 必須参照ではない。
4. 非自明な作業を始める前に active な `Redmine issue / journal と project docs` を確認する。

## Role Boundary

- Claude Code is the default implementer for normal development tasks.
- Codex is the default auditor / policy framer / user-facing clarification actor.
- Standard audit granularity is the UserStory (central preset `### US-Level Audit Model`): Codex audits the whole US — child Task / Test / Bug issues, commits, docs, journals, residual risks — when the US-level Review Request (US audit request) gate is recorded. Per-task Codex review is the exception (guardrail / workflow / release / credential / destructive / migration / architecture / CI-packaging changes, implementer judgment doubt, or an explicit owner / auditor request). Codex audit approval is not close approval; owner close approval stays a separate journal.
- Codex must not directly edit any of the following surfaces unless a Redmine `codex_direct_edit` gate journal exists on the active issue and lists the exact paths under `allowed_paths`:
  - Implementation files: `src/**`, `tests/**`, `docs/**`, `vibes/docs/**`, `README.md`, `RELEASE_NOTES.md`, release workflow, or CLI behavior.
  - Guardrail / custom-instruction / scaffold-rule surfaces: `AGENTS.md`, `CLAUDE.md`, `.mozyo-bridge/rules/**`, `.mozyo-bridge/docs/catalog.yaml`, `.codex/skills/**`, `.claude/skills/**`, `skills/mozyo-bridge-agent/**`, `plugins/mozyo-bridge-agent/**`, scaffold packaged preset / router templates under `src/mozyo_bridge/scaffold/presets/**`.
- Repo-Local Guardrail Autonomous Lane (Redmine #10338, mozyo-bridge product-wide policy distributed via `redmine-governed` / `redmine-rails-governed` presets): `vibes/docs/rules/**`, `vibes/docs/logics/**`, `vibes/docs/specs/**`, `.mozyo-bridge/docs/catalog.yaml` are a Codex-autonomous carve-out from the `codex_direct_edit` gate. Codex may edit these paths directly without a pre-edit gate journal; record a `codex_autonomous_edit` journal (lane / changed_paths / intent / verification / commit_hash / follow_up_review_required) with the commit instead. Detail: central preset `### Repo-Local Guardrail Autonomous Lane` and `vibes/docs/rules/codex-autonomous-guardrail-lane.md`. The carve-out does NOT extend to `AGENTS.md`, `CLAUDE.md`, `.mozyo-bridge/rules/**`, skills, plugins, scaffold preset templates, `src/**`, or `tests/**` — those surfaces still require the `codex_direct_edit` gate.
- A `codex_direct_edit` gate requires `role: 実装者`, `direct_edit: true`, `allowed_paths`, `reason`, and `follow_up_review` recorded as a Redmine journal on the active issue. A chat-level "user said it's fine" or "Codex でやって" is NOT by itself authorization for the gated surfaces; it is an intent to have the work done, not an instruction to bypass Claude.
- `.mozyo-bridge/docs/file_conventions.generated.yaml` and other catalog generator outputs are generator-only artifacts. Even Claude must not hand-edit them; regenerate via `mozyo-bridge docs generate-file-conventions` and verify with `--check`.
- Short imperative phrases such as `進めて`, `対応して`, `実行して`, `go ahead`, `please do it`, `ガードレール変更を明示`, or `Codex direct edit でやって` (without the gate journal) are not by themselves authorization to bypass the role boundary on gated surfaces. When in doubt, fall back to Claude handoff and let the implementer record Implementation Done Gate + Review Request Gate. For autonomous-lane paths Codex may proceed without a chat preamble and record `codex_autonomous_edit` instead.
- Gate-less Codex commits to the gated surfaces above (e.g. `AGENTS.md`, `CLAUDE.md`, `.mozyo-bridge/rules/**`, `README.md` edited directly from a chat instruction) are recorded as correction journals in Redmine and routed through Claude implementation + Codex review on rerun. Autonomous-lane commits are exempt from this correction flow but lacking the `codex_autonomous_edit` journal triggers a follow-up record-keeping correction.

## Verification And Commit

- Use focused `python -m unittest ...` targets when `pytest` is unavailable. Do not install packages just to run tests unless the user approves.
- For scaffold changes, run `mozyo-bridge scaffold status --target .` before commit.
- For governed catalog changes, run `mozyo-bridge docs validate --repo .` and the relevant generator / coverage checks when `catalog.yaml` exists.
- Run `git diff --check` or `git diff --cached --check` before commit.
- Commit only files in the active Redmine issue scope. Leave unrelated dirty files untouched.
- Commit messages for Redmine-scoped work include `Refs: Redmine #<issue_id>` and `issue_<issue_id>`.
- After committing, record the commit hash in the same Redmine issue journal.
