mozyo-bridge workflow command as-is

現状の workflow / handoff / project-gateway コマンド体系を、通常運用で AI が何を見てしまうかという観点で整理する。 これは to-be 設計ではなく、#12755 Add role-based workflow dispatch command without pane-id selection の前提となる as-is 説明である。

一言でいう現状

現状は「標準入口」「専門 primitive」「delivery rail」「debug pane I/O」が同じ CLI 階層に見えている。 そのため AI が本来 mozyo-bridge に任せるべき route / pane / rail 選択を、自分で判断し始める余地がある。
route は意味的に解ける

project-gateway route-plan や consult は semantic identity を使う方向に進んでいる。

入口が分裂している

project-gateway、handoff、q-enter、delegate-* を利用者が選ぶ形になっている。

pane / rail が漏れている

%pane、--target、--mode が通常候補に見え、debug と標準運用の境界が薄い。

as-is の無理が出る箇所

問題 具体例 起きる失敗
AI が command family を選ぶ必要がある consult か child-intake か q-enter かを実行時に探索する。 smoke evidence に runtime command exploration が混ざる。
AI が role 判断を吸収しやすい 親が child に渡す前に domain/design answer や no-dispatch を判断する。 親は交通整理だけという役割境界が崩れる。
pane / rail が表に出る --target %pane、--from-pane、--mode queue-enter を人間や AI が扱う。 semantic identity ではなく pane ID / rail 選択が evidence に見える。
read-only と side-effect が混在する route-plan は read-only、consult は delivery、stamp は mutation。 一歩進めたいだけなのに、どれが安全な一歩かを AI が判断する。
callback が自然文回答と混同されやすい pane 上に返事を書いて止まり、ticketless-callback を呼ばない。 caller lane が durable に結果を受け取れない。

客観的な評価

as-is は部品としてはかなり揃っている。特に semantic route resolution、ticketless callback、 root-to-project consultation、parent-to-child intake はそれぞれ必要な primitive である。 ただし、AI / operator 向けの標準 UX としては command が多すぎる。

問題は primitive の存在ではなく、primitive が通常入口として露出していることにある。 AI に q-enter、queue-enter、%pane、delegate-* を選ばせる限り、 workflow は「mozyo が自動解決する」状態にならない。

as-is / to-be 対比

観点 as-is to-be 狙い
通常入口 project-gateway consult、child-intake、handoff send、q-enter などを状況ごとに選ぶ。 mozyo-bridge workflow step を基本入口にする。 AI / operator が command family を選ばず、mozyo が state から一歩を解決する。
引数 --to、--target、--target-repo、--target-project、--mode、--from-pane などが通常候補に出る。 原則引数なし。必要時だけ --dry-run、--json、--confirm。 role / pane / rail / project identity の解決を CLI 側へ閉じ込める。
route 解決 AI が route-plan をいつ使うか判断する。 workflow step が current lane role と durable gate から必要な route-plan を内部実行する。 route 解決を事前手順ではなく state machine の一部にする。
祖父→親 project-gateway route-plan + project-gateway consult。 workflow step が parent gateway を解決し、ticketless consultation を送る。 祖父は project-domain 判断をせず、routing だけを安全に進める。
親→子 project-gateway route-plan --from-role project_gateway + project-gateway child-intake。 workflow step が child coordinator route を解決し、ticketless work-intake を送る。 親が domain/design 判断を吸収せず、交通整理だけ行う。
子→孫 子が anchor を確保後、handoff send --source redmine --kind implementation_request を選ぶ。 workflow step が anchor 有無を検査し、anchor ready の場合だけ worker dispatch primitive を内部実行する。 worker dispatch の Redmine anchor 必須条件を CLI が fail-closed に守る。
callback handoff ticketless-callback と handoff q-enter --intent consultation_callback を選ぶ。 workflow step が pending callback を検出して標準 callback rail を内部実行する。 pane 上の自然文回答で停止する失敗を減らす。
pane ID %pane が --target や --from-pane として通常説明に出る。 pane ID は cache / self-fence / debug evidence に限定し、route authority にしない。 semantic identity による fail-closed routing を徹底する。
delivery rail queue-enter、standard、pending を利用者が見て選べる。 rail は workflow step 内部の transport decision。通常 help では前面に出さない。 AI が raw Enter / marker / rollback を判断しない。
debug command read、message、type、keys が同じ top-level に並ぶ。 debug/operator namespace または help 上の明確な debug 分類へ退避する。 標準 delivery と pane mutation を混同しない。
失敗時 AI が別 command family を探索し、raw pane 操作や debug delivery に流れやすい。 workflow step が blocked / anchor_required / ambiguous と次 owner を返して止まる。 失敗時も evidence を汚さず、durable record に戻す。

to-be への示唆

as-is の自然な整理先は、通常 AI が mozyo-bridge workflow step だけを叩く形である。 step は現在の lane / Redmine gate / route identity / pending callback を読み、 安全に進められる 1 step だけ進め、判断が必要なら fail-closed で止まる。
to-be 階層 役割 as-is で吸収されるもの
workflow step AI / operator の標準入口。引数なしを基本にする。 route-plan、consult、child-intake、ticketless-callback、handoff send
workflow status 現在 state と next action の read-only 表示。 個別の route-plan / pane read / manual Redmine sweep の一部
primitive / debug 互換、詳細検証、operator debug。 handoff q-enter、delegate-*、read/type/message/keys
