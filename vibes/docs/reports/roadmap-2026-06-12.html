この期間の主題は workspace 横断の session 基盤 と、その上に建てた ユニット状態コックピット です。「複数 workspace で今どの agent が動いていて、 どこで止まっているか」を発見・観測・ジャンプできるところまでを、既存の best-effort / runtime-first 原則を崩さずに積み上げました。本レポートは RELEASE_NOTES.md の Unreleased 節 と事実を整合させています。

Redmine #11628

macOS readdir は NFD、文書・Redmine・agent 経由のパスは NFC で、同じ日本語ディレクトリから別々の session 名が導出されていた。path identity を NFD に固定する共有 helper を入れ、session 名 hash・handoff の --target-repo gate・inventory の registry 照合をすべて同じ正規化に通す。

Current 直近実装 / 設計確定

ユニット状態コックピット（OTel イベントストア + 三層 join）
Redmine=誰の番か / OTel ストア=動いているか / tmux=生きているか、の三層を join して表示する中核機能。4 段階で実装済み：
  • 段階1 OTel イベントストア: agent CLI が直接発行する OpenTelemetry を localhost で受ける自前 OTLP/HTTP 受け口と SQLite single-writer ストア。公式 Collector は立てない。受動・帯域外で agent 本体に波及しない。プロンプト本文は保存せず usage / イベント種別のみ allowlist 保存（本文系 key は deny 優先）。
  • 段階2 inventory への activity 統合: session bootstrap で OTel endpoint と join 属性（mozyo.session / mozyo.agent / mozyo.workspace_id）を注入し、session list に activity（active / idle / unknown）を additive 表示。沈黙は死亡ではなく tmux 縮退。doctor に受け口 health と観測漏れ検出を追加。
  • 段階3 コックピット Web UI: 同じ daemon が http://127.0.0.1:4318/ に unit 一覧 UI を配信（127.0.0.1 のみ）。状態表示・遷移フィード・Reveal in Finder・ジャンプ（tmux switch-client）。action は明示クリックのみ、Content-Type / per-process token / loopback Origin の三重ゲート、UI は DOM API のみで描画。
  • 段階4 Redmine gate 表示: 環境変数で渡した信頼 Redmine base URL と API key で、各 unit の最新更新 open issue を読み取り専用表示。送信先は環境変数 URL に固定（repo 側設定で送信先を変えられない）。Redmine へ書き込まず、ハートビートを残さない。
  • launchd 常駐: otel launchd install / status / restart / uninstall（macOS）。plist に環境変数を書かないため secret が乗らない。
Redmine #11639
operator QoL（tmux UI パック）
iTerm2 の OS window title（session 名ベース）、<prefix> f での Finder ジャンプ、tmux extended-keys 透過（Shift+Enter 改行）、役員向け端末セットアップ runbook。あわせて handoff の --target 'session:window名' 形式が常に失敗していたバグを修正。
Redmine #11638 / #11666
managed desired-state event log の設計確定
「mozyo が何を作ろうとしたか（desired state）」は replay 可能な event log として正本化してよく、「今 生きているか（observed liveness）」は live tmux 正本のまま、という 4 層モデルを設計 doc 化。registry=identity / event log=desired / live tmux=liveness・handoff / inventory=projection の境界を明文化。managed marker は registry anchor 一次 + tmux user option 二次（session 名 prefix は権限境界にしない）。
Redmine #11695
managed marker / desired-state append PoC
marker 分類（anchor 一次 / tmux option 二次 / unmanaged 共存）と、append-only managed-events.sqlite v1（pane_id identity key、socket 拡張点、書込時 NFD 正規化、single-writer、best-effort append）の最小 PoC。liveness / handoff には一切触れない（guard test で import 不在を pin）。
Redmine #11698
managed desired-state event の command-boundary 配線
PoC の append surface を、mozyo が managed pane を作る唯一の境界（new_agent_session_window / new_agent_window の pane_id 確定直後）に最小配線。created event を best-effort append（pane_id=identity / session=attribute / repo_root=NFD 正規化）し、二次 marker を付与。helper 全体を try/except で囲み、append・marker 失敗が pane 生成を壊さない。liveness / handoff target 解決 / preflight / tmux discovery は live tmux 正本のまま（runtime 正本 module に event log/marker の read path を入れないことを guard test で pin）。古典学派寄りに observable contract（外から呼んで event が残る・失敗時も pane_id を返す・正本境界を侵さない）を検証。US-level Codex audit 承認後 main に merge 済みで US はクローズ。RELEASE_NOTES.md の Unreleased にも反映済み（#11742 release preparation）。クローズ済み
Redmine #11726 / #11727 / #11728
Claude design consultation の運用ルール化
owner / Codex の設計議論を実装担当へ Design Consultation として投げ、実装側から反証・懸念・最小 task 分割を返す進め方を、再現可能な project-local rule に整備。毎回必須にはせず、正本境界・責務境界・security/credential/handoff・後戻りコストが高い場面だけ発火する high-signal gate。発火した場合は Answer / disposition を Review / Close で照合する。

Risks / 判断待ち 残リスク

緩和済み / 締めた境界

  • 正本境界の維持（command-boundary 配線後も保持）: liveness / handoff target 解決 / preflight / tmux discovery は live tmux 正本のまま固定する不変条件。#11726 の command-boundary 配線は append（desired-state 記録）のみで runtime 正本に read path を足しておらず、runtime 正本 module が event log/marker を import しないことを guard test で pin 済み。後続実装でもこの境界を崩さないこと。（#11666 / #11625 / #11726）
  • 外部 tmux 直接操作の扱い: marker 不在の session を「unmanaged 共存」とする方針を採用済み。完全排除はしない。（#11698 owner 採用）

残存 / 判断待ち
