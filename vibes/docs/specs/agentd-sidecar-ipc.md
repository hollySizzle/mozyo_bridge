# Agentd / PTY Sidecar IPC Draft

## Status

- Draft: `v0.1`
- Depends on: `vibes/docs/specs/pty-first-agent-runtime.md`
- Purpose: Python-owned `mozyo-agentd` と Node-owned PTY sidecar の最小契約を定義する

## Scope

この文書が扱うのは、次の local control-plane boundary だけである。

- Python 側 application / daemon logic
- Node 側 PTY runtime process

この文書が扱わないもの:

- ticket system APIs
- `mozyo attach` の表示 format
- provider ごとの output parsing ルール

それらは別 spec に切り出す。

## Goals

PTY behavior を application layer に漏らさず、MVP runtime を成立させる最小能力を定義する。

必要な能力:

- runtime session 作成
- provider process を PTY で起動
- structured prompt input を送る
- raw user input を送る
- cancel / terminate
- resize
- normalized sidecar event を購読
- reconnect 後に recent event を replay

## Non-Goals

sidecar は business state の source of truth ではない。

sidecar が所有してはならないもの:

- ticket progress truth
- prompt turn lifecycle truth
- summary generation truth
- durable orchestration decision

それらは Python 側に残す。

## Process Model

### Recommended MVP Shape

- Python process が `mozyo-agentd` を所有する
- Python が 1 つの長寿命 Node sidecar を child process として起動する
- sidecar は `0..n` 個の PTY-backed runtime session を管理する

これにより「1 agent session ごとに Node process 1個」を避けつつ、境界を明示できる。

### なぜ prompt ごとの helper process ではないのか

- 長寿命 session ではコストが高い
- reconnect / attach replay が難しくなる
- cancel と process ownership が複雑になる

## Transport

### Recommended MVP Transport

sidecar process の stdio 上で newline-delimited JSON を使う。

- Python が Node sidecar を child process として起動する
- Python は stdin に NDJSON request を書く
- Node は stdout に NDJSON response / event を書く
- stderr は sidecar diagnostic のみ

理由:

- cross-platform で最も単純
- 最初から socket lifecycle を持たなくてよい
- child ownership が明快

### Future Transport

将来、必要なら local socket transport を追加する。

例:

- 複数 Python client が sidecar を共有したい
- Python daemon restart をまたいで sidecar を残したい

ただし first cut では不要。

## Message Envelope

すべての IPC message は共通 envelope を使う。

```json
{
  "v": 1,
  "kind": "request|response|event",
  "id": "req_123",
  "type": "runtime.session.create",
  "session_ref": "agt_sess_123",
  "payload": {}
}
```

意味:

- `v`: protocol version
- `kind`: request / response / event
- `id`: request と対応 response の関連付けに使う
- `type`: semantic message name
- `session_ref`: runtime session に紐づく message なら必須
- `payload`: message-specific object

## Identity Model

durable な `agent_session_id` は Python 側が所有する。

sidecar は内部 handle を持ってよいが、authority は Python 側にある。

推奨 id:

- `agent_session_id`: durable application id, generated by Python
- `runtime_session_id`: ephemeral sidecar/runtime id, generated by Node
- `prompt_turn_id`: durable application id, generated by Python

Python は session creation 時に `agent_session_id` を渡し、sidecar event が必ず application state に紐づくようにする。

## Requests

### `runtime.session.create`

新しい PTY-backed provider session を作成する。

Payload:

```json
{
  "agent_session_id": "agt_sess_123",
  "provider": "claude|codex|gemini",
  "workspace_path": "/abs/path",
  "argv": ["claude"],
  "env": {
    "KEY": "VALUE"
  },
  "cols": 120,
  "rows": 40
}
```

注意:

- `argv` は Python が決める。sidecar は provider command を勝手に生成しない
- `env` は Python 側で allowlist 後に渡す
- `workspace_path` は PTY cwd

Success response payload:

```json
{
  "runtime_session_id": "rt_123",
  "pid": 99999,
  "started_at": "2026-05-13T01:23:45Z"
}
```

### `runtime.prompt.send`

application-owned prompt turn を注入する。

Payload:

```json
{
  "prompt_turn_id": "turn_123",
  "mode": "structured|raw",
  "text": "prompt body",
  "trailing_enter": true
}
```

ルール:

- `structured` は application / system 起点の prompt turn
- `raw` は「この文字列をそのまま書く」
- prompt turn の identity は sidecar ではなく Python が決める

### `runtime.input.send`

attach 中の人間入力を注入する。

Payload:

```json
{
  "input_id": "input_123",
  "text": "user keystrokes or line",
  "trailing_enter": false
}
```

これは `runtime.prompt.send` と分ける。

人間入力と orchestrated prompt turn は区別できなければならない。

### `runtime.session.resize`

Payload:

```json
{
  "cols": 140,
  "rows": 32
}
```

### `runtime.prompt.cancel`

実行中 prompt turn の interruption を要求する。

Payload:

```json
{
  "prompt_turn_id": "turn_123",
  "strategy": "sigint"
}
```

sidecar は「今どの prompt turn が active か」を勝手に推測しない。Python が明示して渡す。

### `runtime.session.dispose`

runtime session を終了し、PTY resource を解放する。

Payload:

```json
{
  "reason": "operator_request|session_shutdown|daemon_shutdown"
}
```

### `runtime.events.replay`

reconnect 後に recent sidecar-local runtime event を返す。

Payload:

```json
{
  "after_event_id": "evt_123",
  "limit": 200
}
```

これは recovery helper であって durable history の正本ではない。

## Responses

response は常に次を echo する。

- request `id`
- request `type`
- 必要なら `session_ref`

Success envelope:

```json
{
  "v": 1,
  "kind": "response",
  "id": "req_123",
  "type": "runtime.prompt.send",
  "session_ref": "agt_sess_123",
  "payload": {
    "ok": true
  }
}
```

Failure envelope:

```json
{
  "v": 1,
  "kind": "response",
  "id": "req_123",
  "type": "runtime.prompt.send",
  "session_ref": "agt_sess_123",
  "payload": {
    "ok": false,
    "error_code": "runtime_session_missing",
    "message": "runtime session is not active"
  }
}
```

## Events

sidecar は spontaneous に runtime event を emit する。

これらは最終的な application `AgentEvent` taxonomy ではないが、Python が `capture-pane` のような screen scraping なしに normalize できる粒度である必要がある。

### Required MVP Event Types

- `runtime.sidecar.ready`
- `runtime.session.started`
- `runtime.output.chunk`
- `runtime.output.eof`
- `runtime.process.exited`
- `runtime.input.ack`
- `runtime.cancel.ack`
- `runtime.error`

### `runtime.output.chunk`

Payload:

```json
{
  "runtime_session_id": "rt_123",
  "stream": "stdout",
  "seq": 1,
  "text": "raw terminal output chunk"
}
```

ここでは semantic interpretation をしない。

sidecar の責務は「順序を保って raw chunk を出すこと」である。

### `runtime.process.exited`

Payload:

```json
{
  "runtime_session_id": "rt_123",
  "exit_code": 0,
  "signal": null,
  "ended_at": "2026-05-13T01:24:10Z"
}
```

### `runtime.error`

Payload:

```json
{
  "runtime_session_id": "rt_123",
  "error_code": "pty_spawn_failed",
  "message": "spawn ENOENT",
  "detail": {
    "argv": ["claude"]
  }
}
```

## Buffering Rules

sidecar は session ごとに小さな in-memory ring buffer を持つ。

推奨 MVP サイズ:

- count で 500 event
- bytes は unbounded にしない

用途:

- reconnect / attach warm start
- daemon lifetime 中の debug

これは durable storage ではない。

## Error Model

sidecar は stable error code を返す。

最低限:

- `runtime_session_missing`
- `runtime_session_already_exists`
- `pty_spawn_failed`
- `runtime_not_running`
- `stdin_write_failed`
- `resize_failed`
- `cancel_failed`
- `unsupported_provider`
- `invalid_request`
- `internal_error`

Python はこれを application-level finding に変換する。

## Concurrency Rules

この境界では input race を曖昧にしない。

ルール:

- orchestrated prompt turn の serialize は Python が行う
- sidecar は `runtime.prompt.send` と `runtime.input.send` の両方を受ける
- sidecar は受信順をそのまま preserve する
- 「この write が安全か」の policy 判断は sidecar ではなく Python が持つ

つまり session-level locking は Node ではなく Python 側の責務である。

## Lifecycle Expectations

### Startup

1. Python が sidecar を起動する
2. Python は `runtime.sidecar.ready` を待つ
3. その後に request を送れる

### Shutdown

1. 可能なら active session ごとに `runtime.session.dispose` を送る
2. sidecar を graceful shutdown する
3. hang したら force kill する

## Security Boundary

sidecar は local privileged helper として扱う。

ルール:

- spawn を許可する command は Python が決める
- env forwarding は Python が決める
- sidecar は ticket system credential を自分で読まない
- sidecar は durable ticket/task record を直接書かない
- sidecar stderr log に secret を出さない

## この spec が意図的に未定義のもの

次はこの文書では決めない。

- `runtime.output.chunk` をどう `permission_request` や `prompt_completed` に変換するか
- attach terminal の描画形式
- durable SQLite schema の詳細
- multi-daemon / remote daemon behavior

これらは次の spec に分ける。

## 短い結論

最初に必要なのは「Python がすべての provider と直接話す」ことではない。

次である。

- durable orchestration state は Python が持つ
- PTY mechanics は Node sidecar が持つ
- 初期契約は stdio 上の NDJSON で十分
- policy / ordering / durable interpretation は Python が持つ

この分離により、application layer を tmux や screen state に依存させず、PTY layer を将来差し替え可能にできる。
